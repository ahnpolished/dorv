HUM-1415 Task 2: Regression tests for nested GDoc comment reply push

## Changes Made

### 1. Fixed `normalizeRestThreads` in `apps/extension/lib/github/fetch.ts`
- Replaced flat `comments.filter(c => c.inReplyToId === rootComment.id)` with recursive `collectNestedReplies()`
- This ensures nested replies (replies to replies) are included in the thread's replies array
- The GraphQL path already handled this; the REST fallback was missing nested reply collection
- Existing REST fallback test still passes (no behavior change for direct replies)

### 2. Added 2 new tests in `tests/reply-sync.test.ts`

**Test: "pushes a nested GH reply (reply-to-reply) to Drive when parent reply is already mapped"**
- Sets up root CommentMapping (id=10) + parent reply ReplyMapping (id=11)
- Mocks GitHub to return root(id=10), reply(id=11, in_reply_to_id=10), nested(id=12, in_reply_to_id=11)
- Asserts nested reply mapping is created with correct parent chain:
  - ghParentCommentId=11 (actual GitHub parent)
  - docParentCommentId="doc-c-10" (resolved via ReplyMapping fallback to root)
  - source="github"

**Test: "skips nested GH reply when parent reply's inReplyToId has no mapping in either store"**
- Sets up root CommentMapping but uses inReplyToId=999 (ghost parent)
- Asserts nested reply is skipped (no ReplyMapping created)

## Validation
- 179/179 tests pass (32 test files)
- No regressions in existing tests
- Both new tests exercise the real handler (no duplicated logic)

## Root cause chain discovered
1. `normalizeRestThreads` dropped nested replies from threads → they never reached the sync loop
2. The ReplyMapping fallback in `direct.ts` (line 743-753) was correct but never exercised due to #1
3. Fix: recursive `collectNestedReplies` in REST fallback ensures nested replies reach the sync loop
4. The ReplyMapping fallback in the sync loop correctly resolves parent mappings for nested replies
